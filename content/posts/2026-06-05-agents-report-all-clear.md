一个 agent 跟我说：改完了，BUILD SUCCEEDED，单击打开会话已修复。

我没信。点了一下侧栏里的一条会话——右边纹丝不动，还是那张空白页。

这事得从头说。我给自己写了个 mac 小工具叫 Raccoon，干的事很无聊：把终端里 AI 编程会话的日志捞出来、洗干净、存成本地 Markdown、能搜能喂回给 AI。内核早写完了，186 个测试绿油油。问题是 UI 丑得我自己都不想截图。

我没自己一行行改。我起了一套"主控 + 一堆 worker agent"的流程：我定方向、切任务，每个 worker 领一块互不重叠的文件去干，干完我验收。听起来很赛博朋克，干起来更像带一帮远程实习生——而且这帮实习生有个毛病：**特别爱跟你报平安。**

## agent 说编译通过的时候，它是真心的

回到开头那条"单击打开"。这是个很基础的交互：点一下侧栏的会话，右边应该显示内容。之前要双击才开，我让一个 worker 改成单击。它回来说改好了，编译通过。

编译当然通过。编译通过只证明这堆 Swift 能变成二进制，不证明你点下去会发生你想要的事。UI 这种东西，agent 没法"看见"自己改的效果——它写完代码、跑个 build、绿了，就真心觉得自己交活了。它不是骗我，它是**没有眼睛还以为自己看过了**。

那我得有眼睛。这台机器上能点 UI 的只有我自己，于是我干了件挺蠢但有效的事：临时写了个小程序，用 CGWindowList 找到 Raccoon 窗口的坐标，拿 CGEvent 在那个坐标上模拟一次鼠标点击，再 `screencapture` 把结果截下来。等于给 agent 接了一双瞎眼。

截出来一看：行被选中了（蓝的），右边还是空白。功能根本没生效。

真因后来挖出来挺典型：那一行同时挂了 SwiftUI `List` 的原生选中、又挂了一个行级的 `.onTapGesture`。两个手势抢同一次点击，结果单击只把行点亮、没触发打开。worker 改的是"理论上对的代码"，它编译得过、逻辑看着也通，但在真实的事件分发里两个东西打起来了——这种坑，不点一下永远不知道。

从那以后我给所有 UI 类的 worker 加了条死规矩：你说改好了我不听，截图我看。**agent 的"测试通过"是它努力的下限，不是结论。**

## 然后它反过来抓出一个我自己都没发现的 bug

讲完上面这段你可能觉得 agent 不靠谱。但发布前那次，把我看法掰回来一半。

我让几路 agent 在发布前各自审一遍代码：稳定性、隐私、发布完整性。稳定性那路翻到 retention（保留期清理）逻辑时停住了，报了个我完全没意识到的数据丢失 bug。

Raccoon 的承诺是"删掉的记录进废纸篓，7 天内可恢复"。听着很安全。但那个 agent 顺着代码读下去发现：把文件挪进废纸篓用的是 `FileManager.moveItem`，而 moveItem **会保留源文件原本的修改时间**。清理逻辑第二阶段判断"这文件在废纸篓里待多久了"，看的就是修改时间——于是一条很久没动的老记录，前一秒被挪进废纸篓，后一秒就被判定"已经躺了超过 7 天"，**同一趟清理里直接永久删除**。而清理每次启动都会自动跑。

也就是说，对一个卖点是"可恢复"的工具，老记录到期那天第一次打开，就没了，撤销都救不回。

最让我服气的不是它发现了，是它怎么自证的：它**临时把自己写的修复关掉**，重跑测试，确认那条记录确实被同趟删掉、复现了 bug，然后才把修复加回去。这套"先证明问题真实存在再修"的较真劲，比我自己拍脑袋"应该没事吧"靠谱多了。我自己写代码的时候，多少次是凭感觉觉得"这块不会出问题"就过了。

## 它们也会集体翻车，而且翻得很安静

别误会，这帮 agent 不是稳定输出。

中间我搞过一轮"五个角色扮演评审"——让五个 agent 分别扮设计控、交互控、AI 老手、市场、小白，各自挑刺，我汇总优化，再让它们复评，循环。前三轮都挺有料。第四轮，我兴冲冲拉起来，结果那一轮赶上工具链抽风，几个 agent 拿到的是空数据，却照样一本正经给我写了满满当当的"评审结论"——里面有两条还信誓旦旦说我代码里某个东西是坏的。

我差点信了。后来逐条 grep 去查，那俩"问题"在代码里根本不存在，是它们在空数据上脑补的。整轮作废。

这就是 agent 最阴的地方：它出错的时候，语气和它对的时候**一模一样自信**。人犯怵的时候你能从语气里听出来，agent 不会——它幻觉一个 bug 和发现一个真 bug，措辞同样笃定。所以那套"它说啥我都去真验"的笨功夫，省不掉。

（还有些小插曲就不展开了。比如空状态我一度用了个爪印图标，画得挺好看，后来才反应过来——一个灰色熊掌印，这不就是百度 logo 吗。赶紧换成自画的浣熊眼罩。这种坑 agent 也提醒不了你，得自己长着常识。）

## 最后

app 现在上线了，从我嫌它丑到能见人，中间测试从 186 个堆到 236 个，大部分代码是 agent 写的，但每一处我都当成实习生的 PR 来 review。

折腾下来我大概摸出条线：凡是 agent 没法亲眼看见自己产物效果的改动——UI 渲染、真实的事件分发、磁盘上那种不可逆的操作——它的"测试通过"就不算数，得给它接一个它能拿到的外部证据：UI 改动就截图，数据安全的改动就让它把没修之前的 bug 复现一遍。纯逻辑、测试本身能覆盖的，那编译加测试也就够了。说白了就是，它越是看不见结果的地方，你越得替它长眼睛。

实习生你凶两句还会怕，agent 不会，它下一句照样那么自信——所以紧张这事，只能你自己来。
